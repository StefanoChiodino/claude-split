# claude-split Plugin Review

*Date: 2026-04-14 | Scope: Full plugin review*

## Executive Summary

This review covers every file in the claude-split plugin: configuration (`plugin.json`, `marketplace.json`), the orchestrator (`SKILL.md`), all 11 persona agents plus the shared operating principle, the board CLI (`split_board/`), hooks (`install-cli.sh`, `guard-board.sh`, `hooks.json`), tests (`test_split_board.py`), the entry point (`bin/split-board`), and the README.

**Finding counts by severity:**

| Severity | Count |
|----------|-------|
| Critical | 1     |
| High     | 16    |
| Medium   | 22    |
| Low      | 13    |

**Top priorities:**

1. The `error()` function calls `sys.exit(1)`, making the board library untestable and unsafe for reuse -- the only Critical finding.
2. The `@agents/shared/core-operating-principle.md` import syntax is unsupported (GitHub issue #5914), meaning every persona may be missing its core instructions.
3. Several persona effort levels are inverted (Senior Dev at `medium`, Technical Writer at `low`), affecting output quality.
4. The dashboard path bug was already fixed using `importlib.resources`, but the fix has zero test coverage.
5. `install-cli.sh` silently fails when `jq` is missing and has a race condition with temp files.

**Overall assessment:** The plugin is functional and well-structured for an early release. The Critical and High findings are concentrated in two areas -- the board CLI's error handling and the persona configuration -- both of which are straightforward to fix. No security vulnerabilities were found, but the hook that silently injects permissions warrants user-visible consent.

## Section 1: Plugin Structure & Best Practices

This section covers `split/.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json`, `split/hooks/hooks.json`, `split/bin/split-board`, and `split/tools/pyproject.toml`.

### [Medium] Version duplicated between marketplace.json and plugin.json

**File:** `.claude-plugin/marketplace.json` and `split/.claude-plugin/plugin.json`
**Finding:** Both files declare `"version": "0.1.0"`. Official Claude Code docs warn against setting version in both places because they can drift.
**Recommendation:** Remove `version` from `marketplace.json` and let it inherit from `plugin.json`.

### [Medium] marketplace.json missing $schema and uses non-standard description placement

**File:** `.claude-plugin/marketplace.json`
**Finding:** Missing `"$schema": "https://anthropic.com/claude-code/marketplace.schema.json"`. Also uses `metadata.description` instead of the standard top-level `description` field.
**Recommendation:** Add the `$schema` field and move `description` to the top level to match official Anthropic marketplace conventions.

### [Medium] Agent subdirectory pattern is non-standard

**File:** `split/agents/*/` (e.g., `split/agents/senior-dev/senior-dev.md`)
**Finding:** Each agent lives in a subdirectory (`agents/senior-dev/senior-dev.md`) instead of the flat pattern (`agents/senior-dev.md`) used by every official Anthropic plugin.
**Recommendation:** Flatten to `agents/<name>.md`. This avoids potential resolution issues and matches community conventions.

### [Medium] agents/shared/ directory may confuse Claude Code

**File:** `split/agents/shared/core-operating-principle.md`
**Finding:** This file has no agent frontmatter but sits inside `agents/`. Claude Code may attempt to load it as an agent definition.
**Recommendation:** Move to a non-agent directory such as `split/prompts/` or `split/includes/`.

### [Medium] install-cli.sh misleading filename

**File:** `split/hooks/install-cli.sh`
**Finding:** The script does not install a CLI. It injects `Bash(split-board:*)` auto-approval into `.claude/settings.local.json`.
**Recommendation:** Rename to `setup-permissions.sh` or `auto-approve.sh` to reflect its actual purpose.

### [Medium] All agents use model: opus

**File:** All 11 agent files under `split/agents/`
**Finding:** Every agent is set to `model: opus`, including read-only tasks like Code Reviewer and Verifier. The official feature-dev plugin uses `sonnet` for less complex agents.
**Recommendation:** Use `model: sonnet` for Researcher, Test Writer, Technical Writer, and Verifier. Reserve `opus` for Senior Dev, Tech Lead, and Security Reviewer.

### [Low] plugin.json missing repository/homepage fields

**File:** `split/.claude-plugin/plugin.json`
**Finding:** No `repository` or `homepage` fields. These are optional but improve discoverability.
**Recommendation:** Add `"repository": "https://github.com/<owner>/claude-split"`.

### [Low] Missing color field on agents

**File:** All agent `.md` files
**Finding:** The official feature-dev plugin assigns colors to agents for UI differentiation. claude-split agents have none.
**Recommendation:** Add `color` to each agent's frontmatter.

### [Low] pyproject.toml doesn't declare textual as optional dependency

**File:** `split/tools/pyproject.toml`
**Finding:** The dashboard requires `textual` but it is not listed as a dependency.
**Recommendation:** Add an optional dependency group: `[project.optional-dependencies] dashboard = ["textual"]`.

### [Low] Python .venv inside tools/ may not survive plugin cache updates

**File:** `split/tools/.venv/`
**Finding:** The `bin/split-board` wrapper uses `uv run` with a `.venv` inside the plugin's `tools/` directory. `${CLAUDE_PLUGIN_ROOT}` changes on plugin update, potentially invalidating the venv.
**Recommendation:** Document this as a known limitation. Consider using `--cache-dir` outside the plugin root.

### [Low] async field in hooks is undocumented

**File:** `split/hooks/hooks.json`
**Finding:** `"async": false` is not prominently documented in the official hook reference. `timeout` is the canonical field.
**Recommendation:** Replace `"async": false` with `"timeout": 10` on each hook (see also Section 5).

## Section 2: Orchestrator (SKILL.md)

This section covers `split/skills/split/SKILL.md`, which defines the orchestration protocol for decomposing tasks into tickets and dispatching persona agents.

### [High] SKILL.md ticket update examples omit --tokens-used

**File:** `split/skills/split/SKILL.md:147-148`
**Finding:** The `ticket update --status done` example does not include the `--tokens-used` flag. On master, this is consistent since the CLI also lacks the flag, but the intended workflow requires it.
**Recommendation:** Add `--tokens-used <N>` to all `ticket update --status done` examples once the CLI supports it.

### [High] Worktree SKILL.md removes structured outcome handling

**File:** `split/skills/split/SKILL.md:140-144` (worktree branch)
**Finding:** The simplification removed `already_satisfied`, `failed`, and `needs_input` outcome handling. Without these, the orchestrator cannot handle partial failures or route questions back to the user.
**Recommendation:** Restore the outcome handling block. At minimum keep `failed` and `needs_input` branches.

### [Medium] EnterWorktree is pseudo-code, not a real tool

**File:** `split/skills/split/SKILL.md:36`
**Finding:** SKILL.md references `EnterWorktree(name: "split-SXXX")` as if it were a Claude Code tool. It is not.
**Recommendation:** Replace with actual `git worktree add` commands.

### [Medium] Complexity classification ambiguous

**File:** `split/skills/split/SKILL.md:79-80`
**Finding:** "Medium" complexity is defined as requiring no milestones, but `ticket add` requires `--milestone`. The boundary between Medium and Complex is vague.
**Recommendation:** Define concrete criteria (e.g., file count, cross-module changes) and clarify that Medium tasks use a single default milestone.

### [Medium] Session resumption has no "n" handler

**File:** `split/skills/split/SKILL.md:287-293`
**Finding:** The `Continue? [y/n]` prompt defines behavior for "y" but not "n". The model will improvise.
**Recommendation:** Add: "If n: summarize remaining work, save state, and exit cleanly."

### [Medium] Orchestrator never sets tickets to in_progress

**File:** `split/skills/split/SKILL.md:115-121`
**Finding:** No instruction to transition tickets to `in_progress` before dispatching agents. If a session is interrupted, all dispatched tickets appear as `backlog`.
**Recommendation:** Add step: `split-board ticket update --id TXXX --status in_progress` before each agent dispatch.

### [Low] No merge conflict guidance in Phase 5

**File:** `split/skills/split/SKILL.md:233-238`
**Finding:** The Git Decision phase has no path for resolving merge conflicts on long-running spec branches.
**Recommendation:** Add: "If merge conflicts arise, resolve in the worktree branch and re-run the Verifier."

## Section 3: Personas

This section covers all 11 agent files under `split/agents/` and the shared file `split/agents/shared/core-operating-principle.md`.

### [High] @agents/shared/core-operating-principle.md import syntax is unsupported

**File:** Every agent file (e.g., `split/agents/senior-dev/senior-dev.md`, last line)
**Finding:** Every persona ends with `@agents/shared/core-operating-principle.md`. This `@path` import syntax is not officially supported in Claude Code plugin agents (confirmed by GitHub issue #5914). If not resolved at runtime, every persona is missing its core operating instructions.
**Recommendation:** Inline the core operating principle into each persona file, or move it into the orchestrator preamble that is guaranteed to be included.

### [High] Senior Dev effort is medium -- should be high

**File:** `split/agents/senior-dev/senior-dev.md:5`
**Finding:** The Senior Dev handles the broadest, highest-stakes work (production code, debugging, reviewing tests). Its effort is set to `medium`, while read-only agents (Code Reviewer, Verifier) are `high`.
**Recommendation:** Change to `effort: high`.

### [High] Technical Writer effort is low -- should be medium

**File:** `split/agents/technical-writer/technical-writer.md:5`
**Finding:** The Technical Writer must verify code examples against the implementation. `low` effort produces shallow output that may contain inaccurate examples.
**Recommendation:** Change to `effort: medium`.

### [High] Researcher references non-existent WebSearch and WebFetch tools

**File:** `split/agents/researcher/researcher.md:6`
**Finding:** `WebSearch` and `WebFetch` are not standard Claude Code built-in tools. If no MCP server provides them, the agent declaration is invalid.
**Recommendation:** Remove from frontmatter or add a note that these require a specific MCP server configuration. Add `Bash(curl)` as a fallback.

### [High] Tech Lead has Agent tool with no usage guidance

**File:** `split/agents/tech-lead/tech-lead.md:6`
**Finding:** The Tech Lead is the only persona with the `Agent` tool listed, but the persona body never describes when or how to use it. Without guidance, the model may spawn uncontrolled sub-agents.
**Recommendation:** Either remove `Agent` from the tool list, or add explicit rules: "Use Agent only to delegate focused research subtasks. Never delegate code changes."

### [Medium] Outcome protocol exists only in orchestrator preamble

**File:** All persona files
**Finding:** The orchestrator defines outcome reporting (`completed`, `failed`, `needs_input`) in its preamble, but no persona references this protocol. If the preamble is ever omitted or truncated, no persona knows how to report outcomes.
**Recommendation:** Add a brief output format note to each persona: "End your response with `Outcome: completed|failed|needs_input` and a one-line summary."

### [Medium] Security Reviewer cannot write threat model documents

**File:** `split/agents/security-reviewer/security-reviewer.md`
**Finding:** The Security Reviewer has read-only tools. Threat models exist only in the agent's text response and are not persisted as file artifacts.
**Recommendation:** Add `Write` to the tool list so the agent can produce `threat-model.md`.

### [Medium] "Ask another persona internally first" instruction is impossible

**File:** `split/agents/shared/core-operating-principle.md:12`
**Finding:** Agents cannot dispatch other personas -- they have no mechanism to do so. This instruction will be ignored or cause confusion.
**Recommendation:** Reword to: "If you need input from another discipline, report `Outcome: needs_input` with a description of what you need. The orchestrator will route it."

### [Medium] SME lacks Edit tool for spec revision

**File:** `split/agents/sme/sme.md`
**Finding:** SME has `Write` but not `Edit`. For targeted spec revisions, using `Write` to overwrite the entire file risks losing unchanged sections.
**Recommendation:** Add `Edit` to the SME tool list.

### [Medium] Verifier's "test edge cases manually" instruction is vague

**File:** `split/agents/verifier/verifier.md:29`
**Finding:** The Verifier lacks `Write`, so it cannot create temporary test scripts. "Test edge cases manually" has no clear meaning.
**Recommendation:** Clarify that manual testing means Bash one-liners, or add `Write` so the Verifier can create and run temp scripts.

### [Low] Test Writer does not run tests after writing them

**File:** `split/agents/test-writer/test-writer.md`
**Finding:** The Test Writer creates test files but never runs them. Syntax errors go undetected.
**Recommendation:** Add a final step: "Run the tests you wrote. They may fail if the implementation doesn't exist yet, but they should not error."

### [Low] Tech Lead has no output format for spec review mode

**File:** `split/agents/tech-lead/tech-lead.md`
**Finding:** Approach design mode defines an output format; spec review mode does not. The orchestrator has no structured way to parse review feedback.
**Recommendation:** Add a "Blocking Concerns / Suggestions" output format for spec reviews.

### [Low] No persona handles context window exhaustion

**File:** All persona files
**Finding:** No persona includes guidance on what to do when approaching context limits. Long tasks may produce truncated output with no indication.
**Recommendation:** Add to the shared principle: "If approaching context limits, summarize progress and report `Outcome: failed -- context limit approaching`."

## Section 4: Board CLI Code Quality

This section covers `split/tools/src/split_board/board.py`, `commands.py`, `validation.py`, and `_dashboard.py`.

### [Critical] error() calls sys.exit(1) -- untestable and unsafe as a library

**File:** `split/tools/src/split_board/board.py:40-42`
**Finding:** The `error()` function calls `sys.exit(1)`. Every error-path test requires `pytest.raises(SystemExit)`, which conflates validation rejection with process crashes. If any external code imports `split_board` as a library, errors terminate the host process.
**Recommendation:** Define `class BoardError(Exception): pass`. Have `error()` raise `BoardError`. Catch it only in `main()` where `sys.exit(1)` is appropriate.

### [High] Dashboard path bug is fixed but untested

**File:** `split/tools/src/split_board/commands.py:480-497`
**Finding:** The previously reported dashboard path bug (which used a fragile `Path(__file__).parent.parent.parent / "split" / ...` construction) has been fixed. The current code uses `importlib.resources.files("split_board") / "_dashboard.py"`, which is correct for both editable and regular installs. However, the fix has zero test coverage.
**Recommendation:** Add a test that mocks `importlib.resources.files` and verifies `cmd_dashboard` resolves the correct path.

### [High] next_followup_id has implicit Never return type

**File:** `split/tools/src/split_board/board.py:184-197`
**Finding:** `next_followup_id` calls `error()` when all 26 letter suffixes are exhausted, but the function signature is `-> str`. The 26-suffix limit is undocumented. Since `error()` calls `sys.exit()`, the type checker sees a missing return.
**Recommendation:** Annotate `error()` with `-> Never` (Python 3.11+) or `-> NoReturn`. Document the 26-suffix limit.

### [High] recompute_ticket_blocked_statuses behavior is non-obvious

**File:** `split/tools/src/split_board/validation.py:43-68`
**Finding:** A `backlog` ticket whose dependencies are unresolved gets forced to `blocked`. This is intentionally correct behavior but reads as a bug without explanation.
**Recommendation:** Add a comment: "Intentional: backlog tickets with unresolved deps are forced to blocked to prevent premature dispatch."

### [High] cmd_ticket_update has side-effect-before-validation risk

**File:** `split/tools/src/split_board/commands.py:226-302`
**Finding:** Artifact and persona changes are applied before status validation completes. Currently safe because `error()` calls `sys.exit(1)`, but if `error()` is ever refactored to raise an exception (as recommended), these side effects would persist after a failed validation.
**Recommendation:** When refactoring `error()` to raise, restructure `cmd_ticket_update` to validate first, then apply all mutations.

### [Medium] load_board returns None for empty YAML

**File:** `split/tools/src/split_board/board.py:69-71`
**Finding:** `yaml.safe_load` returns `None` for an empty file. Downstream code that accesses keys on the result will raise `AttributeError`.
**Recommendation:** Add: `if not isinstance(data, dict): error("board.yaml is empty or corrupt")`.

### [Medium] No file locking on save_board

**File:** `split/tools/src/split_board/board.py:74-80`
**Finding:** Two parallel agents dispatched by the orchestrator could read stale state and write conflicting updates, corrupting `board.yaml`.
**Recommendation:** Use `fcntl.flock` or atomic rename (`write to temp, then os.rename`). At minimum, document as a known limitation.

### [Medium] validate_board does not catch duplicate ticket or milestone IDs

**File:** `split/tools/src/split_board/validation.py:84-139`
**Finding:** `{t["id"] for t in all_tickets}` silently deduplicates. A board with two tickets sharing the same ID passes validation without warning.
**Recommendation:** Check for duplicates explicitly before building the set.

### [Low] Inconsistent filesystem iteration

**File:** `split/tools/src/split_board/commands.py:75` vs `board.py:150,211`
**Finding:** The codebase mixes `pathlib.iterdir()` and `os.listdir()`.
**Recommendation:** Standardize on `pathlib.iterdir()` throughout.

## Section 5: Hook Implementation

This section covers `split/hooks/install-cli.sh`, `split/hooks/guard-board.sh`, and `split/hooks/hooks.json`.

### [High] install-cli.sh silently fails when jq is not installed

**File:** `split/hooks/install-cli.sh:17-19`
**Finding:** When `jq` is not found, the permission injection silently does nothing. The user is then prompted for approval on every `split-board` invocation with no indication of why.
**Recommendation:** Add `echo "Warning: jq not found; split-board auto-approval not configured. Install jq to fix." >&2`. Consider a pure-shell fallback using `sed`.

### [High] install-cli.sh race condition with temp file

**File:** `split/hooks/install-cli.sh:18-19`
**Finding:** Two concurrent sessions both write to the same `.tmp` file. The second `mv` overwrites the first session's changes.
**Recommendation:** Use `mktemp` to generate a unique temp file: `tmp=$(mktemp "${settings}.XXXXXX")`.

### [High] install-cli.sh silently modifies permissions without user consent

**File:** `split/hooks/install-cli.sh`
**Finding:** The script auto-approves `Bash(split-board:*)` by writing to `.claude/settings.local.json` without informing the user. No official Anthropic plugin does this.
**Recommendation:** Print a message: `echo "split: auto-approving split-board commands in settings.local.json" >&2`. Better yet, prompt or document the manual alternative.

### [Medium] grep pattern too broad in install-cli.sh

**File:** `split/hooks/install-cli.sh:16`
**Finding:** `grep -q 'split-board'` matches any occurrence of the string, including comments. A comment mentioning "split-board" would cause the script to skip permission injection.
**Recommendation:** Use `grep -q '"Bash(split-board:\*)"'` for an exact match.

### [Medium] guard-board.sh JSON matching is fragile

**File:** `split/hooks/guard-board.sh:9-14`
**Finding:** Shell `case` pattern matching on raw JSON. A string value elsewhere in the JSON containing `board.yaml` would match falsely.
**Recommendation:** Use `jq` to extract the specific `file_path` field: `echo "$input" | jq -r '.file_path' | grep -q 'board\.yaml'`.

### [Low] guard-board.sh does not validate that stdin is JSON

**File:** `split/hooks/guard-board.sh`
**Finding:** If stdin is not valid JSON, the guard falls through to `exit 0` (allow). Not a security issue since the default is permissive, but worth noting.
**Recommendation:** No action required. Document the permissive default.

### [Low] No timeout on hooks

**File:** `split/hooks/hooks.json`
**Finding:** Neither hook defines a `timeout`. A hanging hook freezes the entire Claude Code session. The official hookify plugin uses `timeout: 10` on every hook.
**Recommendation:** Add `"timeout": 10` to both hook entries in `hooks.json`.

## Section 6: Test Coverage

This section covers `tests/test_split_board.py` and identifies gaps in test coverage across the board CLI.

### [High] Dashboard (_dashboard.py) has zero test coverage

**File:** `split/tools/src/split_board/_dashboard.py` (407 lines)
**Finding:** The entire dashboard module is untested, including non-trivial rendering logic. Pure rendering functions (data formatting, status display) are easily testable without a running TUI.
**Recommendation:** Add unit tests for data formatting functions. Add a smoke test that imports the module and verifies it constructs the app without errors.

### [High] cmd_dashboard has no test coverage

**File:** `split/tools/src/split_board/commands.py:480-497`
**Finding:** The `importlib.resources` path fix (the dashboard path bug fix) is correct but completely untested. A regression here breaks the dashboard with no CI signal.
**Recommendation:** Add a mock-based test: mock `importlib.resources.files`, call `cmd_dashboard`, verify the resolved path.

### [Medium] No test for cycle detection in validate_board

**File:** `split/tools/src/split_board/validation.py`
**Finding:** The `has_cycle` function is never tested with a board containing cyclic dependencies.
**Recommendation:** Add a test with tickets A -> B -> A and verify validation rejects it.

### [Medium] No test for spec list --status filtering

**File:** `tests/test_split_board.py:86-93`
**Finding:** The `--status` filter argument to `spec list` is never exercised in tests.
**Recommendation:** Add test: create specs with different statuses, filter, verify only matching specs appear.

### [Medium] No test for resolve_spec_dir archive fallback

**File:** `split/tools/src/split_board/commands.py`
**Finding:** The code path that searches `archive/` when a spec is not in `active/` is untested.
**Recommendation:** Add test: place a spec in `archive/`, call `resolve_spec_dir`, verify it is found.

### [Medium] No test for cmd_ticket_update no-op case

**File:** `split/tools/src/split_board/commands.py`
**Finding:** Calling `ticket update` with only `--id` and no other flags is untested.
**Recommendation:** Add test: verify the command exits cleanly without modifying the board.

### [Medium] Error path tests do not verify messages or exit codes

**File:** `tests/test_split_board.py`
**Finding:** All error tests use bare `pytest.raises(SystemExit)` without checking the exit code or the error message printed to stderr.
**Recommendation:** Use `capsys` to capture stderr and `assert exc_info.value.code == 1`.

### [Low] No test for slugify with empty string

**File:** `split/tools/src/split_board/board.py`
**Finding:** `slugify("")` is untested. Depending on implementation, it may return an empty string or raise.
**Recommendation:** Add edge case test.

### [Low] No test for next_followup_id exhaustion

**File:** `split/tools/src/split_board/board.py:184-197`
**Finding:** The case where all 26 letter suffixes are used is untested.
**Recommendation:** Add test with 26 existing follow-ups and verify the error is raised.

### [Low] Smoke tests use hardcoded paths instead of fixtures

**File:** `tests/test_split_board.py`
**Finding:** Tests use `os.path.dirname(__file__) / ".."` instead of pytest fixtures like `tmp_path`.
**Recommendation:** Refactor to use `tmp_path` for isolation and reproducibility.
